## Defects surfaced + fixed

### D11 — LC_LOAD_MAP envelope has no pre-ready buffer

**Severity:** MEDIUM

**Symptom:** If the renderer's bundle takes longer than ~50ms to mount the Svelte component after the bridge subscription installs, an LC_LOAD_MAP envelope emitted by `adapter.loadTimingMap()` is dispatched into the bootstrap's bridge handler but `envelopeHandler` is still `null`, so the load-map is **silently dropped**. The renderer continues displaying the "Waiting for song" placeholder despite the adapter having pushed the map. Subsequent SyncFrames will then arrive and the renderer can't resolve a section, so it stays in the placeholder forever (or shows "No active section" depending on cursor state).

**Root cause:** `OwnWindowOutputAdapter.loadTimingMap()` ([apps/sister/src/output/OwnWindowOutputAdapter.ts:257-274](../../apps/sister/src/output/OwnWindowOutputAdapter.ts#L257)) sends the LC_LOAD_MAP envelope unconditionally as soon as it's called. The adapter has a `#pendingFrames` buffer for SyncFrames pre-ready (line 154-241), but no equivalent for load-maps. `adapter.start()` ([line 166](../../apps/sister/src/output/OwnWindowOutputAdapter.ts#L166)) does not await the renderer-ready signal — it returns as soon as the window is constructed. In `apps/sister/src/main.ts`, `demoEngine.start()` runs immediately after `adapter.start()` resolves and calls `loadTimingMap` synchronously, well before the renderer's bundle finishes loading and signals ready.

**Latency:** Latent since the renderer first started consuming LC_LOAD_MAP envelopes (EP-06 STORY-06.1, this commit). In the current dev environment the bundle is ~17KB and loads inside the SIGTERM-resistant fast path of Electron's file:// loader, so the race never surfaces. Would surface under: slower disk, larger bundle (post-EP-07), CI runners with throttled IO.

**Repro steps:**
1. Inject a 200ms artificial delay at the top of `karaoke-output-bootstrap.ts` (`await new Promise(r => setTimeout(r, 200))`)
2. `LC_DEPLOYMENT_MODE=sister LC_DEMO_MODE=1 electron apps/sister/dist-electron/main.js`
3. Observe: window stays on "Waiting for song" placeholder even though the diag log shows `delivered=120` and frames are flowing through `pushSyncFrame`.

**Evidence:**
- Code-level: [OwnWindowOutputAdapter.ts:262-273](../../apps/sister/src/output/OwnWindowOutputAdapter.ts#L262) sends unconditionally; no equivalent of `#pendingFrames` for the load-map.
- Runtime-level: not directly observable in current demo because the race doesn't fire on this machine; surfaceable via the artificial-delay repro above.

**Fix proposal:** Add a `#pendingLoadMap: LoadMapPayload | null = null` field to `OwnWindowOutputAdapter`. In `loadTimingMap()`, if `!this.#rendererReady`, store the payload and skip the send. In `#onRendererReady()`, send the pending load-map first (before flushing buffered frames) and clear it. Mirror in `ForkOutputAdapter` for symmetry. Add a unit test that calls `loadTimingMap` before the ready latch fires, verifies the envelope is sent on ready.

**Fix status:** **Fixed in commit on this branch.** Applied the proposed fix to `OwnWindowOutputAdapter` and `ForkOutputAdapter`. Five new tests on each adapter (10 total) cover: pre-ready buffering, flush ordering (map first, then frames), last-write-wins for repeated load-maps, immediate delivery after ready, `stop()` clears the buffer. Test sweep 268/268 pass (up from 258).

**Verification:** Live smoke confirms the renderer still receives `LC_LOAD_MAP received for outputId=lyricue-sister-output` after the change. Evidence screenshots regenerated identically.

### D12 — Renderer Content-Security-Policy warning surfaces on every run

**Severity:** LOW

**Symptom:** Electron renderer emits this warning on every `LC_VERBOSE=1` run:
> `[lyricue:sister:renderer:warning] %cElectron Security Warning (Insecure Content-Security-Policy) ...`

The warning is noise but: (1) clutters operator-facing verbose logs; (2) signals that the renderer has no CSP meta tag, which is a real (low-severity) security posture concern even for an offline-only renderer.

**Root cause:** `apps/sister/public/karaoke-output.html` ships without a `<meta http-equiv="Content-Security-Policy" content="...">` tag and the BrowserWindow construction in [electron-browser-window-factory.ts](../../apps/sister/src/output/electron-browser-window-factory.ts) does not call `session.defaultSession.webRequest.onHeadersReceived` to inject a CSP header.

**Latency:** Since `apps/sister` was created (EP-01 STORY-01.2). Has been visible on every verbose run.

**Repro steps:**
1. `LC_DEPLOYMENT_MODE=sister LC_DEMO_MODE=1 LC_VERBOSE=1 electron apps/sister/dist-electron/main.js`
2. Observe stderr: one "Insecure Content-Security-Policy" warning line per renderer-process boot.

**Evidence:** Smoke log at `/tmp/m1-close-smoke.log` (transient). The warning is the only renderer-level warning in the captured output.

**Fix proposal:** Add a `<meta http-equiv="Content-Security-Policy" content="default-src 'self'; style-src 'self' 'unsafe-inline'; script-src 'self'; img-src 'self' data:; connect-src 'none';">` tag to `apps/sister/public/karaoke-output.html`. The renderer loads no external resources and the bundle is self-hosted, so this is a tight CSP. `'unsafe-inline'` for `style-src` is required because the component sets `--progress` and color vars via inline `style=` attributes (architecture.md §4.9 explicitly mandates this pattern for the GPU-composited sweep).

**Fix status:** **Fixed in commit on this branch.** CSP meta tag added with tight defaults: `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; connect-src 'none'; base-uri 'self'; form-action 'none'; object-src 'none';`. Live smoke confirms the "Insecure Content-Security-Policy" warning no longer appears in renderer stderr.
